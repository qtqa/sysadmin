PROJECT DASHBOARD DESCRIPTION:

Level 3: Show Configuration build information

GENERAL
Purpose Improve Continuous Integration (CI) stability, throughput and coverage by fixing Autotests and Configuration builds
Question Is the Configuration build constantly failing, and how Autotest and Configuration build settings (force success, significance) affect to results
 
METRIC
Filters Project, Configuration
Indicator Configuration build information, build history, and list of failed Autotests that caused build failure (significant Autotests in blocking Configurations)
Calculation Build information: Showing data for the build (no calculation)
Build history: Showing a few last Configuration builds with date, result and link to source build results directory (no calculation). By default, the latest Build data is shown, but any listed Build can be selected by clicking the build number. The Timescale filtered and/or the Build selected is indicated.
Failed Autotests: Showing the list of failed significant Autotests if Configuration is not tagged as insignificant i.e. the ones that caused the Configuration build failure.
Interpretation Build information: Note that the build duration may show 00:00:00 for failed builds
Build history: Successive failures or sudden failure after successive successful builds indicate a need for analysis
Failed Autotests: The failed Autotests that caused build failure should be analyzed.
Note that the Autotest dashboard can be utilized here in parallel with the Project dashboard.
Target value Successful builds with not any failed Autotests
 
DATA
Data items Project: e.g. QtBase_stable_Integration
Configuration: several per Project Build (e.g. linux-g++-32_Ubuntu_10.04_x86)
Build number: several per Project (numeric 1 to n; number is Project specific)
Build result: SUCCESS/FAILURE/ABORTED/undef ("undef" means the build was cancelled before it was even started because some other Configuration build failed and caused the whole Project build failure)
Build date: date and time when build started
Build duration: build duration in hh:mm:ss (may show 00:00:00 for failed builds)
Force success: Yes/- (if tagged as "force success" the Configuration build is forced to succeed, so that it will not block the whole Project build; this is e.g. in case of a new or immature Configuration when it is not reasonable to let the whole Project build fail because of this specific Configuration)
Insignificant: Yes/- (if tagged as "insignificant" the Configuration build will not fail even though some significant Autotests would fail)
Autotest name: several per Configuration (e.g. tst_qftp)
Autotest significance: Significant (failure leads to build failure, unless the Configuration set insignificant) / Insignificant (failure does not lead to build failure)
Data source Data in SQL database from log files in http://testresults.qt.io/ci/
Data update cycle New builds are updated to database in real time
